# 铁道车辆智能调度与监控系统 - 产品需求文档 (PRD)

## 1. 项目背景与大方向
本项目基于铁道车辆智能调度与监控的业务流程与专利构思进行开发。
在真实的铁路运输中面临极为复杂的调度任务，尤其是在高峰时段或恶劣天气条件下，传统调度系统依赖静态时刻表，无法满足动态响应需求。
**本项目核心聚焦于参与相关技术/创新比赛**，因此在缺乏真实硬件与传感器数据的客观条件下，系统的重心在于**通过Mock多源随机数据**来构建完整的数据流转闭环，并在**业务验证**与**前端大屏可视化**上做出足够能够证明其实用性和技术含量的展示。

## 2. 核心产品目标
1. **强展示性与可视化**：直观展示列车状态、天气情况的实时变化以及调度中枢的动态决策过程。
2. **逻辑完备的沙盒系统**：打造一个数据采集 -> 处理 -> 智能调度 -> 优先调整 -> 可视化的单向数据闭环以及可交互的动态测试沙盒。
3. **算法流程白盒化体现**：对黑盒的算法模型（如蒙特卡洛模拟或混合序列规划）产生的数据干预过程，通过界面直观反馈（对比：有调度 vs 无调度的不同结果）。

## 3. 用户与角色分析
- **现场评委/演示受众**：重点关注系统呈现的大屏UI视效、突发情况下的“应急响应能力”直观体验。
- **系统调度员 (模拟角色)**：监控全盘情况，并提供人工干预的界面操作（用于比赛互动环节演示）。

## 4. 功能需求分解

### 4.1 数据采集层 (场景与Mock引擎)
因为比赛需要，系统需要内置一个**强大的 Mock 数据生成器引擎**，来代替硬件传感器：
- **常态数据生成**：依照指定频率自动生成列车车速、GPS实时位置更新、车厢红外测量出的乘客流量跳动。
- **气象环境注入**：模拟外部环境数据输入，支持生成冰雪、冻雨概率等随机事件。
- **后台控制台（上帝视角）**：供演讲者/操作人员在比赛演示时，可“一键注入”极端天气、故障或高峰客流，诱发系统的调度反应。

### 4.2 数据处理中心
- **数据清洗仿真**：代码层面展示对不合理数据（例如超过最高安全时速的车速异常包）的主动拦截。
- **实时特征提取器**：通过滑动窗口统计出“车速变化率”、“加速度”、“客流密度”以及“降雪严重度”，这些将作为下游算法决策的核心因子。

### 4.3 智能调度决策中枢
- **业务规则构建**：配置底层约束条件（如列车最小安全防碰撞间距、车站最大停靠容量上限、安全限速）。
- **基础排期演算器**：接收特征，使用混合整数线性规划思想进行发车时间、停靠时间与速度范围的最优化排布。
- **鲁棒性评估体系**：利用蒙特卡洛等模拟方法，结合延误概率，生成一套期望延误最少的基础调度时刻表。

### 4.4 优先级动态重整系统
- **动态事件感知**：当数据层传来突发天气或某一车站客流量骤增时触发策略调整。
- **多维度评分系统**：从“拥挤度”、“行车总延误时长”、“气象恶劣度”进行综合打分，重新排列部分列车的资源使用优先级。

### 4.5 方案生成与控制层
- **冲突校验网关**：将基础调度方案与改写后的优先级进行合并重合，做二次校验，确保在轨道交汇或进站时不发生逻辑性冲撞。
- **终端派发**：把最终确定的安全合法调度时序树通过网络协议（如 WebSocket 或 Server-Sent Events）派发给客户端面板。

### 4.6 可视化交互大屏 (打分重头戏)
- **调度指挥大看板**：科技感强烈、主色调偏暗的UI排版。
- **实时地理/拓扑轨道图**：动态发光的车辆点位沿着轨道线运行，展示车号及当前车速。状态对应颜色（绿：正常，黄：延缓，红：报警）。
- **进出站时间轴列车表**：通过甘特图或滚动时刻表直观反映出各个班次的预计到站时刻调整，演示出“调度前”与“智能调度后”消除延误的对比效果。
- **报警弹窗交互**：监测到恶劣天气导致列车减速时，提供屏幕级报警闪动并在右侧形成处置日志流水单。

## 5. 比赛展示建议与流程（Demo剧本设想）
为了打动评委，整个演示建议分为三幕：
- **第一幕（平稳运行）**：启动系统，屏幕显示大量列车根据初始排期良好运转，指标健康。
- **第二幕（突发事件）**：演讲人通过隐藏后台，拉高站点X的旅客数量，同时设置沿途A-B路段持续大雪。
- **第三幕（智能接管）**：此时屏幕出现报警，某车次降速，眼看将发生产量堆积及大面积延误。这时候“智能调度决策”接管，在大屏上直观看到后续列车改变停发时间优先级，成功消化了延误风险，系统恢复平顺。
